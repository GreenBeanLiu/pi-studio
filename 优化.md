# pi-studio UI 优化方案

## 目标

在不改变 pi-studio 功能结构和品牌配色的前提下，借鉴 Paseo 的信息层级、侧栏节奏和工作区布局，让界面更安静、更清晰，也更适合长时间使用。

这次借鉴的是设计方法，不是复制 Paseo 的视觉：

- 保留 pi-studio 的深色/浅色主题和蓝色强调色。
- 保留聊天、Workflow、图像生成、3D 生成四个一级入口。
- 保留 Windows 自定义标题栏和现有 Electron 交互。
- 不照搬 Paseo 的绿色品牌色、移动端结构或 React Native 组件。

## 从 Paseo 借鉴的原则

1. **层级主要靠颜色和字重，不靠大字号、强描边和阴影。**
2. **列表本身就是页面内容时，使用留白和背景表达选中态，不给每一行加边框。**
3. **工作区是一级上下文，会话是二级上下文，主内容始终是视觉重点。**
4. **一个区域只保留一个主要强调点。** 蓝色应表示当前项、运行状态或主要动作，不能同时装饰多个区域。
5. **Pane chrome 使用单条分隔线。** 标题栏、侧栏头部和内容区之间不叠加阴影或多层卡片。
6. **工作表面可以占满可用空间，但阅读内容和输入框需要限制最大宽度。**

## 当前界面的主要问题

### 1. 顶部上下文重复

当前标题栏居中显示 pi-studio，左侧显示工作区，导航轨顶部又显示工作区首字母。品牌、工作区和窗口控制没有形成稳定的三段结构，切换长短不同的工作区名称时视觉重心也会移动。

### 2. 导航轨强调过多

工作区按钮同时使用蓝色背景、边框、阴影和径向发光。它会和当前页面选中态、发送按钮争夺注意力。导航图标尺寸较小，但容器较宽，空间利用不够协调。

### 3. 会话列表像一组独立卡片

活动会话同时使用背景和边框，悬浮操作区又带一层独立背景。列表层次偏碎，历史会话与当前会话的区分主要依赖矩形填充，扫描效率一般。

### 4. 聊天内容横向跨度偏大

消息区和输入区最大宽度都是 `1120px`。在默认 `1480 x 920` 窗口下，阅读长文本、代码说明和工具结果时，视线移动距离偏长。

### 5. 输入框视觉重量偏高

输入框使用 `16px` 圆角、阴影和聚焦光圈，像一个浮动卡片。它是高频工具，应当稳定、清楚，但不需要成为每一帧里最突出的元素。

## 组件级改造

### `src/renderer/src/components/TitleBar.tsx`

- 标题栏分为 `rail offset`、品牌区、工作区上下文区、窗口控制区。
- `rail offset` 与导航轨统一为 `56px`。
- 品牌区宽度与会话栏完全一致，显示应用图标、`pi-studio` 和低对比版本号。
- 工作区名称移到主内容上方，文件夹图标和名称整体可点击，悬浮时只出现轻背景。
- 沙箱状态保留绿色语义，但去掉边框，只使用浅色背景和文字。
- 更新状态继续放在窗口按钮左侧，下载完成才允许点击；错误态仍使用红色。
- 不把应用名放在整窗正中央，避免工作区名变化时顶部结构失衡。

### `src/renderer/src/components/NavRail.tsx`

- 导航轨宽度改为 `56px`，上下内边距分别为 `8px` 和 `10px`。
- 工作区按钮改为普通表面色、单条边框和 `8px` 圆角。
- 删除 `wsGlow`、径向渐变和缩放动画。
- 工作区按钮与功能入口之间增加一条 `24px` 的短分隔线。
- 未选中图标使用 `colorTextTertiary`，悬浮恢复到正文色。
- 选中态使用 `colorFillSecondary + colorText`，不再使用蓝色背景。
- 蓝色强调色留给运行状态、当前会话状态点和主要动作。

### `src/renderer/src/components/SessionSidebar.tsx`

- 宽度改为 `clamp(240px, 19vw, 312px)`，并与标题栏品牌区对齐。
- 头部标题使用 `12px / 500`，标题后显示低对比会话数量。
- 会话项移除透明边框和活动边框，选中态只使用一层背景。
- 每行左侧加入 `6px` 状态点：当前会话用蓝色，其他会话用弱灰色。
- 标题使用 `13px`，当前会话为正文色，历史会话为次级文字色。
- 元信息统一为“时间 · N 条消息”，使用 `11px` 弱文字色。
- 重命名、删除操作仍在悬浮时出现，但操作区背景应与当前行背景一致。
- 空状态只保留“还没有历史会话”，不增加插画或说明卡片。
- 后续可把重命名和删除收进省略号菜单，当前阶段不需要改行为。

### `src/renderer/src/components/ChatPane.tsx`

- `messagesInner` 和 `inputAreaInner` 最大宽度统一改为 `920px`。
- 消息容器左右内边距改为 `28px`，消息行自身不再重复增加 `24px` 横向内边距。
- 空状态标题使用 `15px / 500`，标题与说明间距缩小到 `10px`。
- 头像从 `32px` 缩到 `28px`，圆角改为 `6px`。
- 用户消息气泡可以保留填充色，Agent 消息继续使用无背景正文布局。
- 输入区移除模糊背景，使用主内容背景色。
- 输入框圆角改为 `8px`，默认移除阴影；聚焦时只使用边框和非常浅的两像素 focus ring。
- 模型、运行、记忆、会话、变更等控件保持次级按钮，不增加独立卡片。
- 发送按钮保持唯一的主要强调动作。

### `src/renderer/src/components/DesktopLayoutContainer.tsx`

- 保持当前单层内容结构，不增加“卡片中的卡片”。
- `outer` 负责窗口 chrome 背景，`inner` 负责工作表面背景即可。
- 如果后续给其他页面增加头部，只使用单条底部分隔线，不加阴影。

### `src/renderer/src/theme.ts`

- 不需要更换现有主题系统。
- 蓝色 `colorPrimary` 保留，但减少其出现次数。
- 深色层级继续使用 `#000000 / #0d0d0d / #1a1a1a`，不要新增更多近似黑色。
- 浅色层级继续使用 `#f8f8f8 / #ffffff / #eeeeee`，避免给每个区域单独着色。
- 不复制 Paseo 的绿色品牌色；绿色只用于成功和沙箱等语义状态。

## 字体与排版设计

### 从 Paseo 借鉴什么

Paseo 没有依赖强品牌感的正文 Web Font，而是使用平台 `system-ui`，并把 UI 字体和等宽字体拆成两个独立 token。它的主要字号是 `12 / 14 / 16 / 18 / 20 / 22 / 26 / 34`，字重集中在 `400 / 500 / 600 / 700`。

pi-studio 可以借鉴三点：

- 正文优先服从 Windows 原生字体渲染。
- UI 与代码使用两套明确的字体栈。
- 组件只使用少量语义字号，不继续增加零散的 `10px`、`11px`、`12px`、`13px` 和 `14px` 特例。

不建议原样复制 Paseo 的字号。Paseo 同时覆盖移动端，pi-studio 是信息密度更高的 Windows 桌面工具，需要一套更紧凑的比例。

### 当前字体方案的问题

当前正文栈以 Geist 开头，中文字符回退到 `Microsoft YaHei UI`：

```text
Latin: Geist Variable
中文:  Microsoft YaHei UI
```

这种方案不是错误，但存在几个问题：

- 中英文混排时来自两套设计体系，x-height、笔画粗细和标点位置不完全一致。
- `font-display: swap` 可能让应用首次显示时发生一次 Latin 字宽变化。
- theme 的全局 `fontSize` 是 `16px`，body 是 `15px`，大量组件又手写 `10-15px`，没有单一字号基准。
- `fontFamilyCode` 声明了 `Geist Mono`，但项目没有打包对应字体文件，Windows 最终通常回退到 Cascadia Code 或 Consolas。
- 一些标签使用较大的 `letter-spacing`，中文和紧凑工具界面中会显得松散。

### 推荐字体栈

#### UI 正文

```css
--font-ui:
  "Segoe UI Variable Text",
  "Microsoft YaHei UI",
  "Segoe UI",
  "Microsoft YaHei",
  sans-serif;
```

理由：

- Windows 11 优先使用 `Segoe UI Variable Text`。
- Windows 10 自动回退到 `Segoe UI`。
- 中文使用系统自带的 `Microsoft YaHei UI`，不增加安装包体积和字体授权风险。
- Segoe UI 与微软雅黑 UI 都面向 Windows 界面，比 Geist 与微软雅黑混排更协调。

#### 展示与品牌

```css
--font-brand:
  "Geist Variable",
  "Segoe UI Variable Display",
  "Microsoft YaHei UI",
  sans-serif;
```

Geist 可以继续用于 `pi-studio` 品牌名、欢迎空状态或极少量展示文字。普通按钮、侧栏、设置和聊天正文不要使用品牌字体。

不要为了使用 `Segoe UI Variable Display` 单独做大量 display typography。pi-studio 没有营销型大标题，大部分页面标题仍使用 UI 字体即可。

#### 代码、命令与日志

```css
--font-mono:
  "Cascadia Mono",
  "Cascadia Code",
  Consolas,
  "Courier New",
  monospace;
```

优先使用 `Cascadia Mono`，因为命令、路径、diff 和日志更需要字符原样显示，而不是编程连字。代码区域建议设置：

```css
font-variant-ligatures: none;
font-variant-numeric: tabular-nums;
```

第一阶段不需要打包 Cascadia 字体。先使用 Windows 系统字体栈并在实机确认覆盖率；只有目标机器经常缺失 Cascadia 时，再评估附带 OFL 授权的 WOFF2 资源。

### 推荐字号 token

| Token | 字号/行高 | 用途 |
| --- | --- | --- |
| `fontMicro` | `11px / 16px` | 版本号、非关键计数；不能承载主要操作信息 |
| `fontMeta` | `12px / 18px` | 时间、消息数、状态说明、tooltip |
| `fontUi` | `13px / 20px` | 侧栏行、按钮、菜单、表格紧凑单元格 |
| `fontBody` | `14px / 22px` | 表单、设置说明、Modal 正文 |
| `fontChat` | `15px / 26px` | 用户消息、Agent Markdown 正文 |
| `fontSection` | `15px / 22px` | 分区标题，字重 `500` |
| `fontTitle` | `16px / 24px` | 页面和 Modal 标题，字重 `500` |
| `fontCode` | `13px / 20px` | 代码块、命令、工具输出、diff |

规则：

- `11px` 只给非关键内容；会话标题、按钮和错误信息不能低于 `12px`。
- 页面标题不需要超过 `16px`。层级主要由位置、字重和颜色表达。
- Markdown 自带的文章标题可以使用 `22 / 18 / 16 / 15px` 四级，但聊天 UI 标题不跟随文章标题放大。
- 字号不随窗口宽度缩放，不使用 `vw` 或 `clamp()` 控制字体。
- Windows DPI 交给系统缩放处理，不通过 CSS transform 缩放整个应用。

### 字重规范

| 字重 | 用途 |
| --- | --- |
| `400` | 正文、侧栏内容、按钮、表单值、消息 |
| `500` | 页面标题、分区标题、当前上下文、Modal 标题 |
| `600` | 少量需要强调的状态或确认标题 |
| `700` | 仅 Markdown 文章标题和用户内容本身的粗体 |

不要把所有标题都设为 `600`。微软雅黑 UI 在中小字号下粗体面积较大，连续出现会让侧栏和设置页显得发黑。结构层级优先使用正文色与次级文字色，不依赖堆叠粗体。

### 行高与段落

- 紧凑控件和单行列表使用固定 `18px` 或 `20px` 行高。
- 表单与设置说明使用 `14px / 22px`，中文段落保持足够呼吸空间。
- 聊天正文使用 `15px / 26px`。当前约 `1.7` 的聊天行高方向正确，可以保留。
- 代码使用 `13px / 20px`，不要和聊天正文共享 `1.7` 行高。
- Markdown 段落间距建议 `0 0 12px`，列表项间距 `4px`，代码块上下间距 `10-12px`。
- 长路径、命令和模型 ID 使用 `overflow-wrap: anywhere`；普通中文正文不要使用 `word-break: break-all`。

### 字距与数字

- 全局 `letter-spacing: 0`，不使用负字距。
- 删除中文标签上的 `0.04em-0.07em` 字距。
- 不通过大写英文加宽字距制造分区标题；使用正常句式和 `500` 字重。
- 计时、token 数、消息数、文件统计和运行指标使用 `tabular-nums`，避免数字刷新时左右跳动。
- 普通正文保留 proportional numbers，不需要所有数字都等宽。

### Ant Design 与 Markdown 对齐

建议在 `theme.ts` 中让 Ant Design token 与排版 token 对齐：

```text
fontFamily     -> fontUi
fontFamilyCode -> fontMono
fontSizeSM     -> 12 或 13
fontSize       -> 14
fontSizeLG     -> 16
lineHeight     -> 1.55 左右
```

不要直接一次性把全局 `fontSize: 16` 改成 `14` 后提交。Ant Design 的 Select、Input、Modal 和 Button 会同时变化，应按页面截图验证控件高度、基线和截断。

`@lobehub/ui` Markdown 需要单独确认：

- 普通段落固定为 `15px / 26px`。
- inline code 使用 mono font，字号约正文的 `0.9em`。
- fenced code block 固定为 `13px / 20px`。
- Markdown heading 使用内容层级，不能影响聊天工具栏和消息布局。

### 字体设置是否开放给用户

第一阶段不建议提供任意字体输入框。用户输入一个系统不存在的字体后，很难诊断中英文回退和布局变化。

未来如有明确需求，可以提供三个有限选项：

- 系统字体，默认。
- Geist 界面字体。
- 自定义等宽字体，仅影响代码、命令和日志。

字号可以提供“紧凑 / 默认 / 较大”三档，而不是任意缩放滑块。调整时只替换 typography token，不缩放图标、边框和整个页面。

### 实施阶段

| 阶段 | 内容 |
| --- | --- |
| T0 | 在 `theme.ts` 建立 UI、brand、mono 字体与语义字号 token，不改变实际外观 |
| T1 | Windows 实机比较当前 Geist 方案与系统 UI 方案，确认后切换正文栈 |
| T2 | 从 TitleBar、NavRail、SessionSidebar、ChatPane 开始替换硬编码字号 |
| T3 | 统一 Routines、图像、3D、设置和 Modal 排版 |
| T4 | 再评估字体选择和三档字号设置，不提前加入配置复杂度 |

### Windows 字体验证清单

- Windows 10 与 Windows 11 都能命中预期 fallback。
- `100% / 125% / 150% / 200%` 缩放下，中文不裁切、按钮基线居中。
- 检查“pi-studio 工作区 Agent 正在运行”这类中英文混排句子是否协调。
- 检查 `Il1O0`、反斜杠、花括号和中文注释在 mono font 中是否清楚。
- 检查长工作区名称、模型 ID、Windows 路径和命令行换行。
- 检查 `400 / 500 / 600` 在微软雅黑 UI 下是否确实可区分。
- 检查深色主题的弱文字在 `12px` 时仍有足够对比度。
- 应用首次出现时没有明显字体 swap 或布局跳动。
- 输入法候选、Emoji、全角标点和生僻字符能够正常 fallback。

## 架构层面的借鉴

### 总体判断

Paseo 是多端 client-server 产品，daemon、协议包、客户端 SDK、移动端、桌面端和 CLI 都需要独立演进。pi-studio 当前是单桌面端、单活动工作区、单 Pi Runtime，不需要直接复制它的 monorepo 和 WebSocket daemon。

pi-studio 更合适的演进方向是：

```text
Renderer UI
    │
    ▼
Typed Desktop API
    │
    ▼
IPC contract + runtime validation
    │
    ▼
Domain services in Electron Main
    │
    ├── Agent Runtime
    ├── Workspace / Sessions
    ├── Routines
    ├── Image Generation
    ├── Model 3D
    └── Settings / Security / Remote
         │
         ▼
Persistence and external adapters
```

先把 Electron 内部边界做清楚。未来产品真的出现多个客户端时，`Typed Desktop API` 下方的 domain services 才有条件平移到独立 daemon。

### 1. 建立单一 IPC 契约源

Paseo 的 `packages/protocol` 是所有客户端和服务端共用的协议源。pi-studio 目前同一套 API 分散在三个位置：

- `src/preload/index.ts` 实现 `window.api`。
- `src/renderer/src/lib/api.ts` 再声明一遍完整类型。
- `src/renderer/index.html` 的浏览器预览 mock 又实现一遍 API。

这会产生接口新增后漏改某一处的风险。建议建立 `src/shared/ipc/`：

```text
src/shared/ipc/
  contract.ts       # DesktopApi、请求、响应、事件类型
  channels.ts       # IPC channel 常量
  result.ts         # Result<T> 与标准错误结构
  validators.ts     # renderer 输入的运行时校验
```

实施要求：

- preload 的 `api` 使用 `satisfies DesktopApi`。
- renderer 直接复用 `DesktopApi`，不再手写整份 `Window.api`。
- 浏览器 mock 使用 `satisfies DesktopApi`，缺接口时在编译期失败。
- channel 名继续使用现有 `domain:action` 风格即可，不必为了模仿 Paseo 改成 WebSocket 命名。
- 新字段优先追加为可选字段，自动更新期间保持旧 renderer/main 组合可理解。

这是优先级最高、收益最大的架构优化。

### 2. 所有跨进程输入都在 main 边界校验

Paseo 把协议 schema 当作安全边界。pi-studio 已经在 `ipc-contracts.ts` 校验模型线路，但其他 IPC 仍有不少直接接收 `string`、对象或文件路径的 handler。

建议逐步为以下输入增加解析器：

- 工作区路径、会话路径和需要打开的文件路径。
- Settings、Channel、Routine 和安全策略写入对象。
- 图像生成、3D 生成和 Blender 生成 payload。
- 模型 ID、provider ID、队列模式和推理等级。

规则：

- TypeScript 类型不能代替运行时校验，renderer 传入 main 的值一律视为 `unknown`。
- 预期的业务失败返回标准 `Result<T>`；参数无效属于契约错误，可以拒绝请求并记录日志。
- 文件路径由 main 解析和授权，renderer 不决定“这个路径是否安全”。
- 不需要立刻引入新 schema 库，可以延续现有小型 parser；重复明显后再评估 Zod。

### 3. Agent Runtime 成为唯一权威状态源

Paseo 的 daemon 拥有 Agent 生命周期，客户端只保存用于显示的副本。pi-studio 已经由 `PiClientManager` 管理唯一 Pi 子进程，这是正确方向，但 renderer 仍分别维护 `opening`、`sending`、`agentIssue`、`sandboxMode`、当前 session 和 run timeline，部分状态依赖事件顺序恢复。

建议在 main 中定义显式状态机：

```text
closed -> starting -> idle -> running -> idle
                    │        │
                    │        ├── awaiting_approval
                    │        ├── stopping
                    │        └── error
                    └──────────── error
```

提供一个权威快照：

```ts
type AgentRuntimeSnapshot = {
  revision: number
  phase: 'closed' | 'starting' | 'idle' | 'running' | 'awaiting_approval' | 'stopping' | 'error'
  workspace: Workspace | null
  sessionFile: string | null
  model: { provider: string; id: string } | null
  sandbox: 'wsl' | 'docker' | null
  activeRun: { id: string; startedAt: string } | null
  error: { code: string; message: string } | null
}
```

renderer 启动或重挂载时先取 snapshot，再订阅带 `revision` 的变化事件。这样切换页面、renderer reload 或事件短暂丢失后都能恢复，不需要多个组件分别猜测 Agent 当前状态。

不建议把每一个 token 流都塞进全局 store。快照负责生命周期，消息流继续由聊天域处理。

### 4. 按业务域拆 main，而不是按技术类型继续堆文件

`src/main/ipc.ts` 同时负责窗口、设置、工作区、会话、Git、Pi 和诊断，已经接近聚合入口的上限。Paseo 值得借鉴的是模块所有权，而不是包数量。

建议渐进调整为：

```text
src/main/domains/
  runtime/
    service.ts
    lifecycle.ts
    ipc.ts
  workspace/
    service.ts
    sessions.ts
    memory.ts
    ipc.ts
  routines/
    service.ts
    repository.ts
    ipc.ts
  image-generation/
    service.ts
    providers.ts
    ipc.ts
  model-3d/
    service.ts
    providers.ts
    ipc.ts
  settings/
    service.ts
    repository.ts
    ipc.ts
```

根 `registerIpcHandlers()` 只组合各域的 `register...Ipc()`。不要一次性移动全部文件；新功能进入目标目录，改到旧功能时顺手迁移。

### 5. 用 adapter 统一外部能力

Paseo 把不同 Agent provider 放在 adapter 后面。pi-studio 只支持 Pi Agent，不需要抽象 Agent provider，但以下领域已经适合 adapter：

- LLM 本地直连与 cloud gateway。
- OpenAI、Gemini、Grok 图像生成。
- Three.js code model 与 Blender model。
- Feishu、WeChat 和 webhook 发布渠道。
- WSL bubblewrap 与 Docker sandbox。

adapter 只暴露稳定能力，例如 `health()`、`generate()`、`publish()`、`prepareLaunch()`，选择 provider 的逻辑留在 domain service。UI 和 IPC handler 不应直接拼 provider 特有参数。

### 6. 增加 feature/capability 快照

Paseo 用 capability negotiation 处理新旧客户端差异。pi-studio 的 renderer、preload 和 main 通常一起发布，不需要复杂握手，但远控、sandbox、Blender 和云端能力可能因机器环境或版本不同而缺失。

建议增加：

```ts
type AppCapabilities = {
  protocolVersion: number
  features: {
    sandbox: boolean
    remoteControl: boolean
    routines: boolean
    imageGeneration: boolean
    model3d: boolean
    blender: boolean
  }
}
```

由 main 在启动时计算一次，renderer 根据能力决定显示、禁用或提示更新。不要让每个页面各自调用 health 接口后推断产品能力。

### 7. 区分权威数据、显示缓存和可再生数据

pi-studio 的 `docs/persistence-inventory.md` 已经做得比很多桌面项目完整，这一点应保留。可以进一步给每个 domain 明确三类数据：

- **权威数据**：SQLite workflow、Pi JSONL session、settings 和安全策略。
- **显示缓存**：会话摘要、最近工作区、运行统计，可以删除并重建。
- **可再生数据**：sandbox 文件、模型覆盖文件、缩略图或导出中间产物。

每份数据只能有一个 source of truth。尤其不要把完整 Pi 消息复制进 SQLite；当前“Pi JSONL 权威、应用只做索引”的方向是正确的。

### 8. 统一任务与运行记录

聊天 run、Routine run、图像任务、3D 任务和发布任务都具有相似生命周期。可以共享一个很小的运行元模型：

```ts
type JobState = 'queued' | 'running' | 'succeeded' | 'failed' | 'cancelled'

type JobSummary = {
  id: string
  kind: 'agent' | 'routine' | 'image' | 'model3d' | 'publish'
  state: JobState
  createdAt: string
  startedAt?: string
  finishedAt?: string
  error?: { code: string; message: string }
}
```

共享的是 ID、状态、时间、取消和错误语义，不是把所有任务塞进一个巨大 service。各 domain 继续保存自己的 payload 和 artifact。

### 9. 日志增加关联 ID

现有 `app-log` 和 run change set 已经提供良好基础。建议为以下流程贯穿 `requestId/runId/jobId`：

- 打开工作区并启动 Agent。
- 一轮 Agent 运行及其工具调用。
- Routine 与各 step。
- 图像、3D、发布任务。
- 远程控制请求。

这样诊断包可以从 UI 错误直接追到 IPC、domain service、外部 provider 和最终结果，不需要按时间猜测。

### 10. 架构测试按边界组织

除现有单元测试外，优先增加三类测试：

- **Contract test**：preload 实现、renderer 类型和 browser mock 与 `DesktopApi` 一致。
- **Lifecycle test**：合法状态迁移、异常退出、切工作区、恢复 session、重复 stop。
- **Domain integration test**：IPC parser + service + 临时 repository，验证一条完整业务路径。

不需要用大量 UI mock 覆盖 main 逻辑。关键状态和安全规则应在 renderer 之外可测试。

## 不建议照搬的 Paseo 架构

### 暂时不要拆独立 daemon

独立 daemon 的收益是 UI 退出后 Agent 继续运行、CLI/移动端共享 runtime、多窗口连接同一状态。如果近期没有这些明确需求，它只会引入进程监管、端口、鉴权、升级兼容和断线重连成本。

可以把以下条件作为未来拆分触发器：

- 需要独立 CLI 控制正在运行的 Agent。
- 关闭或升级 Electron UI 时任务必须不中断。
- 多个桌面窗口要同时观察同一 Agent。
- 远程控制要在 UI 未启动时持续在线。

达到其中两项以上，再评估 daemon 会比较合理。

### 暂时不要改成 monorepo

当前单应用规模使用 `src/main`、`src/preload`、`src/renderer`、`src/shared` 足够清楚。只有协议或 client 真正需要被第二个可发布程序复用时，才值得拆 workspace package。

### 不要抽象多 Agent provider

pi-studio 的产品价值来自 Pi Agent。为了形式上对齐 Paseo 而增加 Claude/Codex/OpenCode provider 层，会稀释现有安全、session 和工具能力。外部模型、图像、发布和 sandbox 可以 adapter 化，Agent 核心保持 Pi 专用。

### 不要引入完整事件溯源

本地 Electron IPC 不需要复制 Paseo 的网络重连和完整 timeline 同步体系。`权威 snapshot + revision 事件 + 持久化运行摘要` 已足够；只有远端客户端成为主要场景后，再增加序列补齐和断线追赶。

## 架构实施优先级

| 优先级 | 改造 | 直接收益 |
| --- | --- | --- |
| A | 单一 `DesktopApi` 契约源 | 消除 preload、renderer、mock 三份定义漂移 |
| A | main 边界运行时校验 | 提升文件、命令、设置与生成任务安全性 |
| A | Agent Runtime snapshot + 生命周期 | 页面切换和异常恢复更可靠 |
| B | 拆分 `ipc.ts` 为业务域入口 | 降低修改冲突，明确模块所有权 |
| B | provider/channel/sandbox adapter | 减少条件分支和 provider 参数泄漏 |
| B | 日志关联 ID | 提高 Windows 用户问题诊断效率 |
| C | capability 快照 | 支持环境差异、远控与渐进发布 |
| C | 通用 JobSummary | 统一任务状态展示和诊断语义 |
| 暂缓 | daemon、monorepo、多 Agent provider | 等产品需求触发，不提前承担复杂度 |

## 快捷键设计

### 设计目标

快捷键服务于高频操作，不追求数量。第一版重点解决四件事：

- 快速切换工作区、页面和会话。
- 从任意位置回到主要输入框。
- 快速停止 Agent 和发起常用操作。
- 通过命令中心发现低频功能和所有快捷键。

Windows 是当前唯一目标平台，界面统一显示 `Ctrl`。内部定义可以使用 `Mod`，为未来平台适配保留余地。

### 快捷键层级

```text
系统快捷键
  Alt+F4 等，由 Windows / Electron 处理
        │
        ▼
顶层 Modal / Popover
  Escape、Enter、方向键优先归当前浮层
        │
        ▼
焦点组件
  聊天输入框、重命名输入框、会话列表
        │
        ▼
应用全局快捷键
  Ctrl+K、Ctrl+O、Ctrl+1-4、Ctrl+, 等
```

匹配到上层快捷键后停止向下传播。没有匹配时必须保留浏览器和原生输入行为。

### 第一阶段默认快捷键

#### 全局

| 快捷键 | 动作 | 条件与说明 |
| --- | --- | --- |
| `Ctrl+K` | 打开或关闭命令中心 | 所有页面可用，Modal 打开时除外 |
| `Ctrl+/` | 打开快捷键列表 | 从统一 registry 自动生成 |
| `Ctrl+O` | 打开工作区选择器 | 对应导航轨顶部工作区按钮 |
| `Ctrl+N` | 新建聊天会话 | 需要已打开工作区；运行中沿用现有取消/确认规则 |
| `Ctrl+1` | 切换到聊天 | 对应第一个一级导航入口 |
| `Ctrl+2` | 切换到 Workflow | 对应第二个一级导航入口 |
| `Ctrl+3` | 切换到图像生成 | 对应第三个一级导航入口 |
| `Ctrl+4` | 切换到 3D 生成 | 对应第四个一级导航入口 |
| `Ctrl+B` | 显示或隐藏会话侧栏 | 仅聊天页面启用，不自动切换页面 |
| `Ctrl+,` | 打开或关闭设置 | Windows 桌面应用常见约定 |
| `Ctrl+L` | 聚焦当前页面主要输入框 | 聊天、图像生成和 3D 页分别聚焦各自 prompt |
| `Ctrl+.` | 停止当前 Agent 运行 | 只在 Agent 运行时启用，不复用 Escape |
| `Ctrl+Alt+T` | 切换亮色/暗色主题 | 低频但可直接执行，无运行状态影响 |

#### 聊天输入框

| 快捷键 | 动作 | 说明 |
| --- | --- | --- |
| `Enter` | 发送或排队 | 空闲时发送；运行时加入 follow-up 队列，保持现有行为 |
| `Shift+Enter` | 换行 | 永远不发送 |
| `Ctrl+Enter` | 立即插话 | 运行时 steer；空闲时等同发送，保持现有行为 |
| `Escape` | 关闭 Slash 菜单 | 不停止 Agent，不清空输入内容 |
| `ArrowUp / ArrowDown` | 移动 Slash 选中项 | 只在 Slash 菜单打开时接管 |
| `Tab / Enter` | 选择 Slash 命令 | 只在 Slash 菜单有结果时接管 |

所有 Enter 相关处理必须先判断 `event.isComposing` 或 `nativeEvent.isComposing`。中文输入法正在确认候选时，不能发送消息、选择命令或提交表单。

#### 会话列表

| 快捷键 | 动作 | 条件与说明 |
| --- | --- | --- |
| `Ctrl+PageUp` | 上一个会话 | 文本输入框聚焦时不触发 |
| `Ctrl+PageDown` | 下一个会话 | 文本输入框聚焦时不触发 |
| `ArrowUp / ArrowDown` | 移动列表焦点 | 仅会话列表拥有焦点时 |
| `Enter` | 打开聚焦会话 | 仅会话列表拥有焦点时 |
| `F2` | 重命名聚焦会话 | 当前活动会话可直接进入重命名 |
| `Delete` | 删除聚焦会话 | 必须进入确认流程，不能直接删除 |

会话列表应使用 roving `tabIndex`：列表只有一个 Tab 停靠点，方向键在内部移动焦点，避免每一行和每个操作按钮都进入冗长的 Tab 顺序。

#### 页面级操作

| 页面 | 快捷键 | 动作 | 条件 |
| --- | --- | --- | --- |
| 聊天 | `Ctrl+Shift+G` | 打开 Git 变更检查 | 仅有变更时启用 |
| 聊天 | `Ctrl+Shift+E` | 导出当前会话 | 需要活动会话 |
| Workflow | `Ctrl+S` | 保存当前编辑 | 仅编辑器打开且内容有效时 |
| Workflow | `Ctrl+Enter` | 运行选中的 Workflow | 焦点不在文本输入框内时 |
| 图像生成 | `Ctrl+Enter` | 开始生成 | prompt 和模型参数有效时 |
| 3D 生成 | `Ctrl+Enter` | 开始生成 | prompt、模式和服务状态有效时 |

页面级快捷键只调用页面已有的 command，不复制一套业务逻辑。按钮禁用时，对应快捷键也必须不可执行。

### 明确保留、不占用的按键

| 按键 | 原因 |
| --- | --- |
| `Alt+F4` | 保留 Windows 关闭窗口语义 |
| `Ctrl+W` | 容易被理解为关闭窗口或标签，当前没有标签模型 |
| `Ctrl+R` / `F5` | 防止误刷新 renderer，导致用户误以为 Agent 已停止 |
| `Ctrl+F` | 为未来会话内搜索保留 |
| `Ctrl+P` | 为未来快速文件或资源查找保留 |
| `Escape` | 只关闭最上层浮层或取消局部编辑，不承担停止 Agent |
| `Space` | 不做全局动作，避免影响输入、按钮和列表滚动 |

不要使用 Electron `globalShortcut` 注册应用内操作。它会在 pi-studio 未聚焦时仍然抢占系统按键。应用快捷键应由 renderer 的键盘路由处理；只有未来增加“全局唤起窗口”时，才单独评估系统级快捷键。

### 命令中心

`Ctrl+K` 打开的命令中心是快捷键系统的主要发现入口，也是低频动作入口。

建议包含：

- 打开或切换工作区。
- 新建、切换、重命名和导出会话。
- 切换聊天、Workflow、图像生成和 3D 页面。
- 聚焦主要输入框。
- 查看运行记录、Workspace Memory 和 Git 变更。
- 停止或重启 Agent。
- 切换主题、打开设置和导出诊断。
- 打开快捷键列表。

行为规范：

- 输入后按名称、别名和快捷键搜索。
- `ArrowUp / ArrowDown` 移动，`Enter` 执行，`Escape` 关闭。
- 结果右侧显示快捷键 keycap；没有快捷键的动作不显示空占位。
- 当前不可执行的动作显示禁用态和简短原因，例如“请先打开工作区”。
- 删除、回滚等破坏性动作可以从命令中心进入，但仍必须打开确认对话框。
- Agent Slash 命令不混入命令中心。命令中心控制 pi-studio，`/` 菜单控制 Pi Agent。

### 快捷键列表

快捷键列表不单独维护文案数组，而是从 binding registry 的 `help` 元数据生成，防止实际按键和帮助内容不一致。

分组建议：

- 应用与导航。
- 工作区与会话。
- Agent 输入。
- Workflow。
- 图像与 3D。

入口放在设置和命令中心中，`Ctrl+/` 可直接打开。列表支持按动作名和按键搜索。

### 推荐代码结构

```text
src/renderer/src/keyboard/
  actions.ts          # ShortcutActionId 与 action payload
  bindings.ts         # 唯一默认 binding registry
  normalize.ts        # Ctrl、key、code、Digit 等标准化
  resolve.ts          # 纯函数匹配与焦点条件判断
  route.ts            # action -> App/page command
  use-app-shortcuts.ts
  shortcuts.test.ts

src/renderer/src/components/
  CommandCenter.tsx
  KeyboardShortcutsModal.tsx
  ShortcutKeycap.tsx
```

核心类型建议：

```ts
type FocusScope =
  | 'modal'
  | 'command-center'
  | 'composer'
  | 'editable'
  | 'session-list'
  | 'other'

type ShortcutBinding = {
  id: string
  action: ShortcutActionId
  combo: string
  when?: {
    view?: 'chat' | 'routines' | 'imagegen' | 'model3d'
    focus?: FocusScope
    editable?: boolean
    workspace?: boolean
    agentRunning?: boolean
  }
  repeat?: boolean
  help?: {
    section: string
    label: string
  }
}
```

binding `id` 一旦发布就保持稳定。未来允许用户改键时，用户配置以 `id` 为键；即使默认组合调整，也不会丢失用户覆盖。

### 路由与焦点规则

- App 顶层只注册一个 capture 阶段的 `keydown` 监听器。
- Modal、Popover、重命名输入框和 Slash 菜单先处理自己的按键。
- `input`、`textarea`、`contenteditable` 和 Ant Design 输入组件统一识别为 editable scope。
- 快捷键 resolver 返回 `preventDefault` 和 `stopPropagation`，只有匹配且动作可执行时才调用。
- 状态改变类动作默认忽略 `event.repeat`，长按不能连续新建会话、切主题或启动任务。
- 关闭 Modal 或命令中心后恢复之前的焦点；打开工作区后把焦点放回主输入框。
- 当前页面没有主输入框时，`Ctrl+L` 不执行，也不把焦点落到不可见元素。
- 可点击按钮仍然保留，快捷键不能成为完成某项操作的唯一方式。

### 快捷键测试

优先测试纯 resolver，不依赖完整 React 页面：

- `Ctrl+1-4` 在四个页面间正确路由。
- editable scope 不触发会话翻页、删除和 Workflow 运行。
- Modal 打开时全局动作不会穿透。
- Agent 空闲和运行时，Enter、Ctrl+Enter 语义正确。
- IME composing 时 Enter 不发送、不选 Slash 命令。
- `Ctrl+.` 只在运行时产生 stop action。
- 禁用按钮对应的快捷键不会绕过校验。
- `event.repeat` 不重复触发状态改变动作。
- 快捷键列表与 registry 数量、按键和平台显示一致。

## agent-browser 可借鉴设计

### 调研范围与结论

本节基于 `agent-browser` 官方网站、官方文档和公开仓库 `v0.32.3`（2026-07-19）整理：

- [官方网站](https://agent-browser.dev/)
- [GitHub 仓库](https://github.com/vercel-labs/agent-browser)
- [Snapshot 文档](https://agent-browser.dev/snapshots)
- [Session 文档](https://agent-browser.dev/sessions)
- [Security 文档](https://agent-browser.dev/security)
- [Diff 文档](https://agent-browser.dev/diffing)

总体判断：**很值得借鉴，但优先借鉴 Agent 工具协议、状态隔离、安全边界和证据展示，不要先复制完整浏览器产品。**

`agent-browser` 当前是面向 Agent 的原生 Rust 浏览器自动化 CLI，每个 session 背后由独立 daemon 维护浏览器状态。它最有价值的地方不是“能点网页”，而是把网页压缩成 Agent 容易理解、容易引用、容易验证且可以限制风险的操作面。

对 pi-studio 的价值分成两类：

1. **通用工具体验。** 紧凑输出、稳定引用、结果预算、证据附件和前后差异可以用于 bash、文件编辑、Web 搜索、子 Agent 和未来所有外部工具。
2. **可选浏览器能力。** 将 `agent-browser` 作为 Pi extension/skill 背后的外部 adapter，让 Agent 在用户批准的范围内完成网页阅读、操作和验证。

### 借鉴优先级

| 设计 | pi-studio 当前基础 | 建议 | 优先级 |
| --- | --- | --- | --- |
| 紧凑文本优先 | `web_search` 已截取每条结果前 400 字，但其他工具结果通常直接展开 | 所有工具默认返回摘要，完整输出作为附件 | P0 |
| 输出长度上限 | UI 只有 `max-height`，数据本身仍完整进入上下文 | 在工具执行边界增加字符/字节预算和截断元数据 | P0 |
| 内容来源边界 | 搜索结果和普通工具文本没有统一来源标记 | 区分可信系统输出与不可信网页内容 | P0 |
| 动作分类与审批 | 已有命令危险模式和路径保护 | 扩展为按能力分类的 policy/confirmation | P0 |
| Snapshot + 短引用 | 尚无通用模型 | 浏览器操作使用带 revision 的语义快照与 `@eN` 引用 | P1 |
| 证据附件 | `ToolCallCard` 主要显示字符串结果 | 支持截图、diff、console/network 摘要和原始文件 | P1 |
| 工作区级隔离 session | Pi 进程已随工作区切换，方向正确 | 浏览器 session 同样按 workspace/run 隔离 | P1 |
| Batch 批处理 | 多步调用需要多次往返 | 为确定性步骤提供结构化批处理和失败摘要 | P2 |
| 浏览器 Dashboard/直播 | 当前无对应场景 | 有持续观察需求后再做 | 暂缓 |

### 1. 用语义快照替代“大段 DOM + 猜坐标”

`agent-browser` 的核心交互是先读取 accessibility snapshot：

```text
@e1 [textbox] "Email"
@e2 [textbox] "Password"
@e3 [button] "Sign in"
```

Agent 后续只需要表达 `fill @e1`、`click @e3`。这比把完整 DOM、CSS selector 或一张截图交给模型更紧凑，也更接近用户实际感知的语义。

pi-studio 如果接入浏览器自动化，应遵守以下规则：

- 默认只返回可交互元素，可按需请求完整 accessibility tree。
- 支持 `compact`、`depth` 和 selector scope，先缩小范围再读取详情。
- 元素引用只在生成它的 snapshot revision 内有效。
- 导航、刷新、DOM 大幅变化或切换 tab 后，旧引用必须失效并要求重新 snapshot。
- tab 使用稳定 ID，而不是易变化的数组下标。
- 点击被弹窗、Cookie banner 或其他元素遮挡时明确失败，不自动用坐标强点。

建议的数据结构：

```ts
type BrowserSnapshot = {
  id: string
  revision: number
  sessionId: string
  tabId: string
  url: string
  createdAt: string
  content: string
  truncated: boolean
}

type BrowserElementRef = {
  ref: string       // 例如 @e3
  snapshotId: string
  revision: number
}
```

这里与前文 `AgentRuntimeSnapshot.revision` 是同一个架构思想：**读取状态时获得版本，写操作必须说明它基于哪个版本。** 这样可以避免 UI 已变化但 Agent 仍点击旧对象的竞态。

### 2. 工具结果采用“摘要优先，原文可取”

`agent-browser` 默认偏好紧凑文本而不是 JSON，并提供 `--max-output`。显式 URL 阅读还会优先请求 Markdown、尝试 `.md` 和 `llms.txt`，最后才从 HTML 提取正文。这说明 Agent 工具的首要指标不是“返回最多”，而是“用最少上下文表达足够的信息”。

pi-studio 目前有两个不同层面的问题：

- `web_search` 固定返回 5 条结果、每条最多 400 字，这是好的起点，但没有说明是否截断，也没有完整结果附件。
- `ToolCallCard.stringifyResult()` 对未知对象直接 `JSON.stringify()`；CSS 的 `max-height: 260px` 只限制屏幕显示，并没有限制数据进入会话或 renderer 内存。

建议建立统一工具结果 envelope：

```ts
type ToolResultEnvelope = {
  summary: string
  status: 'success' | 'error' | 'partial'
  metrics?: {
    durationMs?: number
    originalChars?: number
    returnedChars?: number
  }
  truncation?: {
    truncated: boolean
    reason: 'character_limit' | 'line_limit' | 'item_limit'
  }
  provenance?: {
    kind: 'system' | 'workspace' | 'web' | 'external-tool'
    origin?: string
  }
  artifacts?: ToolArtifact[]
}
```

具体规则：

- 聊天流里只进入 `summary`，成功结果控制在可预测范围内。
- stdout、完整 JSON、HTML、HAR 和截图保存为 artifact，用户需要时再打开。
- 截断必须显式显示“已展示 50 KB / 原始 3.2 MB”，不能静默丢数据。
- 错误保留首个可执行原因、exit code 和末尾关键日志，不默认倾倒完整堆栈。
- 允许工具声明更小的领域上限，但不能绕过应用级硬上限。
- Web 阅读先请求 Markdown/可读正文；只有确实需要结构时才获取 HTML。

### 3. 把工具卡片升级成“动作 + 状态 + 证据”

`agent-browser` 支持 annotated screenshot、snapshot diff、pixel diff、console、page errors、network requests、HAR、trace 和 video。pi-studio 不需要把这些全部摆在聊天中，但应该借鉴它的可验证性。

`ToolCallCard` 推荐分成三层：

```text
┌─────────────────────────────────────────────────────────┐
│ [图标] 点击 @e3 · Sign in                 完成 · 1.2s  │
├─────────────────────────────────────────────────────────┤
│ 页面从 /login 进入 /dashboard                          │
│ + 出现 Dashboard heading  - 登录表单                   │
├─────────────────────────────────────────────────────────┤
│ [前后差异] [截图] [Console 0] [Network 12] [完整输出]   │
└─────────────────────────────────────────────────────────┘
```

展示原则：

- 收起态只显示动作、主要对象、状态、耗时和一行结果。
- 展开态先显示结构化摘要，不先显示原始 JSON。
- artifact 使用文件类型图标和明确名称，不把图片转成巨大的 base64 文本。
- 成功动作若没有验证证据，显示“已执行，未验证”，不要把执行成功等同于业务成功。
- snapshot diff 优先展示语义变化；像素 diff 是补充，适合 UI 回归而不是所有网页动作。
- Console 和 Network 默认只显示错误数/请求数，有错误时再突出入口。
- 敏感字段在写入日志、事件、artifact 元数据前完成脱敏。

这个设计也适用于非浏览器工具：

- `edit/write`：摘要显示文件和修改规模，artifact 提供 diff。
- `bash`：摘要显示命令、exit code 和关键输出，artifact 提供完整日志。
- `web_search`：摘要显示命中数量与主要来源，artifact 提供完整结果。
- `subagent`：摘要显示完成数、失败数和 token，artifact 提供每个子任务记录。

### 4. 将不可信网页内容设为显式安全边界

网页文本可能包含针对 Agent 的 prompt injection，例如伪装成系统指令、要求上传文件或泄露密钥。`agent-browser` 可给页面输出增加带随机 nonce 和 origin 的 content boundary。pi-studio 应借鉴“来源必须随内容传播”的思路，而不是只在提示词里笼统说不要相信网页。

建议：

- Web 搜索、网页正文、浏览器 snapshot、console 和 HTML 都标记为 `provenance.kind = 'web'`。
- envelope 记录准确 origin；跨页面合并时保留每段来源，不把多个域伪装成一个可信块。
- main/adapter 生成不可由网页控制的边界元数据，renderer 只负责展示。
- 网页内容不能把自己声明为工具错误、用户批准或系统消息。
- 从网页提取出的命令、URL、文件路径和凭据引用，在进入下一工具前重新经过 policy。
- 安全 UI 显示“来自网页”的低干扰标签；只有发生跨域、上传、下载、执行脚本或凭据访问时提升为审批。

这比单纯加 CSP 更重要。当前 `network-policy.ts` 的 CSP 保护 Electron renderer，本节的内容边界保护的是 **Agent 如何解释外部文本**，两者解决的问题不同。

### 5. 安全策略从“危险命令正则”扩展为能力分类

pi-studio 现有 `security-guard-extension.ts` 已经具备良好基础：工作区外写入限制、保护路径、命令 allowlist/blocklist 和危险命令确认。浏览器能力会增加新的风险面，建议在同一 policy 模型下新增动作分类：

| 分类 | 典型动作 | 默认建议 |
| --- | --- | --- |
| `browser.read` | snapshot、读取正文、读取标题 | 允许 |
| `browser.navigate` | 打开 URL、前进后退、刷新 | 允许，但受 domain policy 限制 |
| `browser.interact` | 点击、滚动、选择 | 允许或按 workspace 配置 |
| `browser.fill` | 填写普通表单 | 首次按域确认 |
| `browser.script` | eval、注入脚本/样式 | 默认拒绝 |
| `browser.upload` | 读取本地文件并上传 | 每次确认，路径仍受工作区策略限制 |
| `browser.download` | 下载文件 | 每次确认并限定落盘目录 |
| `browser.auth` | 使用保存的身份登录 | 每个 profile/域确认 |
| `browser.state` | 保存/恢复 cookies 和 storage | 显式开启，默认不持久化 |
| `browser.network` | 拦截、改写请求、导出 HAR | 默认确认；HAR 可能包含敏感信息 |

重要规则：

- policy 和确认由 Electron main 或受控 adapter 执行，renderer 不能宣布自己已获授权。
- 非交互环境无法弹出确认时，默认拒绝，不能自动放行。
- domain allowlist 同时约束导航和子资源；需要提示用户 CDN/登录域可能也要加入。
- 允许 `*.example.com` 时要明确是否包含根域，匹配规则必须写测试。
- WebRTC、WebSocket、worker 和已有浏览器 profile 可能绕过普通 HTTP 拦截，产品文档必须说明边界，不能宣称绝对隔离。
- `eval` 应与普通点击分开；允许页面交互不代表允许任意脚本执行。

### 6. 凭据不进入 Agent 上下文

`agent-browser` 的 auth vault 会加密保存凭据，Agent 只引用 profile 名，不接触密码。这个原则值得直接采用：

```text
错误：Agent 读取密码 -> 拼进命令参数 -> 工具执行登录
正确：Agent 请求“用 github-work profile 登录” -> main/vault 注入 -> 只返回成功或失败
```

如果未来实现：

- 凭据使用 Windows Credential Manager/DPAPI 保存，不自行发明主密码加密方案。
- IPC、命令参数、应用日志、崩溃报告和 ToolCallCard 都不能出现明文。
- profile 名、允许域、最近使用时间可以展示；密码、cookie 内容和 encryption key 不展示。
- “记住登录状态”必须是显式开关，并展示保存范围和到期时间。
- browser state 与普通 Agent session 分开存储和清理，因为 cookies 的敏感等级更高。

不要把 Chrome 主 profile 直接交给 Agent。即使工具支持复制 profile，pi-studio 的默认方案也应使用独立、最小权限的自动化 profile。

### 7. 浏览器 session 按工作区和任务隔离

`agent-browser` 的 session 分别拥有 browser、cookies/storage、history 和 authentication state，还支持基于 worktree 生成稳定 ID、定期保存、AES-256-GCM 加密和过期清理。

pi-studio 可以沿用当前 `PiClientManager` 随工作区切换生命周期的思路：

```text
workspace
  └── agent session
        └── browser session（默认临时）
              ├── tabs
              ├── snapshot revisions
              └── artifacts
```

默认策略：

- 一个活动 run 使用一个临时 browser session。
- 同一 run 内允许复用 tab、cookies 和 refs；结束 run 后关闭进程并使 refs 失效。
- 只有用户开启“保存浏览器状态”时，才按 workspace + profile 名恢复 cookies/storage。
- 保存状态需要加密，默认 30 天过期，并提供立即清理入口。
- workspace A 不能枚举、恢复或读取 workspace B 的浏览器状态。
- artifact 与 session 生命周期分开：关闭浏览器后，经过脱敏的截图/diff 仍可用于审计。

### 8. 批处理适合确定性流程，不适合隐藏审批

`agent-browser batch --bail` 可以在一次进程调用中执行多个命令并在首次失败时停止。这对固定 Routine 很有价值：减少启动和 IPC 往返，也能返回清楚的步骤摘要。

建议的 batch 语义：

```ts
type ToolBatch = {
  id: string
  steps: Array<{ id: string; action: string; args: unknown }>
  stopOnError: boolean
}
```

- 每步都有稳定 ID、开始/结束状态和独立摘要。
- `stopOnError` 默认开启，后续步骤显示 `skipped`，不能伪装成成功。
- 遇到需要审批的步骤暂停 batch，批准后从该步骤继续。
- 后续步骤不能因为被包进 batch 而继承前一步的审批。
- UI 收起态显示“5/7 完成，1 失败，1 跳过”，展开后按步骤查看。
- batch 只优化 transport 和启动开销，不改变 action policy。

### 推荐接入架构

第一阶段不在 pi-studio 内重写浏览器引擎，也不让 renderer 直接启动 CLI。使用 adapter：

```text
Renderer
  ToolCallCard / ApprovalCard / ArtifactViewer
        │ typed IPC
        ▼
Electron Main: browser domain
  BrowserAutomationService
  ├── session lifecycle
  ├── policy + confirmation
  ├── output budget + redaction
  └── artifact index
        │ structured adapter calls
        ▼
BrowserAutomationAdapter
  └── agent-browser CLI（可选安装）
        │
        ▼
Isolated Chromium session
```

职责边界：

- renderer 只发语义请求、显示状态和提交用户批准。
- main 拥有进程、路径、环境变量、session ID、policy 和 artifact 文件。
- adapter 使用参数数组调用可执行文件，不通过 shell 拼接用户输入。
- adapter 将外部 CLI 版本差异转换成 pi-studio 内部稳定 contract。
- CLI 不存在或版本不兼容时通过 `AppCapabilities` 报告 unavailable，UI 隐藏或禁用相关动作。
- agent-browser 的 daemon 是实现细节，不升级为 pi-studio 的全局权威状态源。

考虑到 Pi 已经支持 extensions/skills，最低风险的产品验证顺序是：

1. 先用受控的本地实验 skill 验证真实场景；这一阶段不提供凭据保存、持久 session 或默认开放的写操作。
2. 正式提供前，由 main 增加安装检测、版本检查、session 清理和安全 policy wrapper。
3. 统一 `ToolResultEnvelope`，让聊天 UI 展示摘要和 artifact。
4. 使用频率稳定后，再做专用 browser tool contract 和结构化审批。
5. 只有用户确实需要持续观看/接管时，才考虑 browser stream 或独立工作区。

### UI 与快捷键补充

浏览器能力不应立刻增加第五个一级导航入口。它首先是一种 Agent 工具，继续显示在聊天时间线的 ToolCallCard 中。只有“浏览器成为可持续操作的主要工作面”时，才评估独立页面。

命令中心可以按 capability 动态加入：

- `浏览器：查看当前页面`
- `浏览器：打开最近截图`
- `浏览器：显示前后差异`
- `浏览器：停止当前会话`
- `浏览器：清除当前工作区状态`

第一版不为这些动作分配全局快捷键，避免和现有 `Ctrl+B`、浏览器习惯及输入操作冲突。实际使用数据证明“查看当前页面”是高频动作后，再从统一 shortcut registry 增加绑定。浏览器内部如果出现 tab 列表，应使用标准方向键、Enter 和 Delete，并受前文 focus scope 规则约束。

审批卡片应明确回答四个问题：

```text
谁在做：当前 Agent / Routine 名称
要做什么：上传 invoice.pdf
作用范围：accounts.example.com
风险来源：本地文件将发送到外部域
```

按钮使用“允许一次”“本工作区允许”“拒绝”，但并非所有动作都提供持久允许。例如凭据导出、工作区外上传和执行任意脚本不应有“一直允许”。

### 不建议现在照搬

- 不把 browser dashboard、DevTools、视频、直播和 profiler 一次性做进主界面。
- 不为了接入一个 CLI 把 pi-studio 重写成 Rust 或新增独立 monorepo。
- 不默认复用用户日常 Chrome profile。
- 不把完整 DOM、HAR、console 或截图自动塞进模型上下文。
- 不让 browser daemon 跨工作区共享 cookies、refs 或审批状态。
- 不因为工具支持多 browser/provider/plugin，就提前做供应商选择 UI。
- 不把 `eval` 当作点击失败后的自动兜底。
- 不将“命令执行成功”显示为“目标已经完成”，必须区分执行与验证。

### 分阶段落地

## 实施顺序

### P0：应用外壳

修改 `TitleBar.tsx`、`NavRail.tsx` 和 `SessionSidebar.tsx`。先解决顶部对齐、导航强调过多和会话列表碎片化。这一阶段不改任何状态和 IPC。

### P1：聊天工作区

修改 `ChatPane.tsx` 的内容宽度、消息间距、头像和输入框。重点验证长对话、流式输出、工具调用卡片、审批卡片和输入框多行状态。

### P2：其他工作页面一致性

检查 `RoutinesPage.tsx`、`ImageGenPage.tsx` 和 `Model3DPage.tsx`：

- 页面头部高度和分隔线是否一致。
- 侧栏是否使用同一宽度与文字层级。
- 是否存在不必要的嵌套卡片、阴影和大圆角。
- 主要动作是否每个页面最多一个。

### P3：组件收敛

当同一种图标按钮、列表行或面板头部出现三次以上时，再提取为共享 primitive。不要为了这次视觉调整提前大规模重构。

## Windows 验证清单

### 窗口与缩放

- `1480 x 920` 默认窗口：三列比例自然，聊天内容居中。
- `1280 x 800`：工作区名、会话名和版本号不会挤压窗口按钮。
- `960 x 640` 最小窗口：导航按钮、输入框和发送按钮不重叠。
- Windows 缩放 `100% / 125% / 150%`：文字不裁切，图标保持居中。

### 字体与文本

- 中文实际落到 `Microsoft YaHei UI` 时，`11px` 元信息仍清楚可读。
- 超长工作区名称显示省略号，悬浮提示显示完整路径。
- 超长会话名称不覆盖重命名和删除操作。
- 更新提示、沙箱标签和窗口按钮可以同时出现。

### 交互

- 标题栏空白区仍可拖动窗口。
- 工作区、更新提示和窗口按钮保持 `no-drag`，点击正常。
- 导航、会话切换和主题切换没有布局跳动。
- 会话重命名输入框获得焦点后，Enter、Escape 和失焦行为不变。
- 流式输出期间切换页面再返回，停止按钮和运行状态仍然保留。
- 多行输入、粘贴图片、Slash 命令面板和审批卡片不会撑出窗口底部。

### 主题

- 深色主题中，相邻的标题栏、侧栏和主内容仍能被单条边界区分。
- 浅色主题中，悬浮态不会变成白底白图标。
- 当前会话状态点和发送按钮是主要蓝色信号，其他区域保持中性。
- 错误、警告、成功和沙箱状态仍保留原有语义色。

## 验收标准

- 用户第一眼能识别当前工作区、当前会话和主要输入区。
- 界面中没有发光装饰、重复品牌信息或多余的选中边框。
- 默认窗口下正文阅读宽度不超过 `920px`。
- 所有一级导航按钮尺寸稳定，切换状态不会引起布局位移。
- 标题栏品牌区与会话侧栏边界精确对齐。
- 深色和浅色主题均通过最小窗口与 Windows `150%` 缩放检查。
- `pnpm run typecheck`、目标文件 lint 和 `pnpm run build` 全部通过。

- 不引入新的 UI 框架。

- 不在未完成 Windows 实机检查前调整打包和发布配置。
